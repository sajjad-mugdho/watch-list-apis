1. Executive Summary

This documentation provides a complete technical guide for implementing the Dialist user onboarding system. It covers three integrated flows:

📌 Three Onboarding Flows:
1) Authentication - Clerk sign-up/sign-in
2) Platform Onboarding - 4-step profile wizard (all users)
3) Merchant Onboarding - Finix KYC form (sellers only)

Key Features

  • ✅ Single Source of Truth: MongoDB (Clerk is JWT token cache)
  • ✅ Real-time Sync: Clerk ↔ Database via webhooks
  • ✅ Step-by-Step Progress: Users can pause/resume onboarding
  • ✅ Merchant KYC: Finix integration for payment approval
  • ✅ Error Recovery: Retry logic with exponential backoff
  • ✅ Session Management: JWT claims auto-refresh

Platform Support

All endpoints are available for:

Client Type Authentication Supported
Web Application Clerk JWT + Header ✅ Yes
Mobile App (iOS/Android) Clerk JWT + Header ✅ Yes
Desktop Dashboard Clerk JWT + Header ✅ Yes
API Testing (Dev) x-test-user header ✅ Yes

💡 Key Architecture Decision:
MongoDB is the source of truth. Clerk's publicMetadata is a cache only. After any onboarding step, we sync user state to both Clerk (fast) and MongoDB (persistent).

3. Complete User Journeys

Journey 1: Sign-Up & Platform Onboarding (4 Steps)

This is the most common flow. Every new user must complete this before accessing the marketplace.

Journey 2: Merchant Onboarding (Finix KYC)

This flow is optional but required to sell on the marketplace. Users must complete Journey 1 first.

User Clicks "Become a Seller" │ ├─► Check: Has onboarding_status = "completed" ? (REQUIRED) │ If NO → Error: "Complete platform onboarding first" │ └─► POST /api/v1/marketplace/merchant/onboard ▼ Backend Calls Finix API Backend: createOnboardingForm() ├─► Call: Finix.createOnboardingForm() ├─► Finix returns: { form_id, form_link, expires_at } ├─► Create MerchantOnboarding document └─► Return onboarding_url to client ▼ Frontend Redirect Browser: window.location.href = onboarding_url ├─► Redirect to: https://onboarding.finix.com/form/ONF_xxx │ ▼ User Completes Finix Form │ ├─► Enters business details: │ - Business name │ - Business type (sole proprietor, LLC, etc.) │ - Tax ID / EIN │ - Business address │ - Business phone │ ├─► Enters personal details: │ - Full name │ - Date of birth │ - Social security number │ - Address │ ├─► Enters bank account: │ - Routing number │ - Account number │ - Account holder name │ └─► Submits form ▼ Finix Processes & Sends Webhooks Webhook 1: onboarding_form.updated (status: COMPLETED) ├─► form_id matches ├─► Backend: Calls Finix to provision merchant └─► Sets: identity_id Webhook 2: merchant.created ├─► merchant_id assigned ├─► onboarding_state = "PROVISIONING" ├─► Backend syncs to Clerk └─► isMerchant = false (NOT APPROVED YET) ▼ Finix Performs KYC Verification │ ├─► Business verification ├─► Personal identity verification ├─► Bank account verification │ ▼ Finix Verification Complete Webhook 3: merchant.updated (onboarding_state: APPROVED) ├─► Finix sets merchant verification status ├─► Backend updates MerchantOnboarding ├─► Sets: isMerchant = true ├─► Sets: verification_state = "SUCCEEDED" ├─► Syncs to Clerk └─► Backend: fires event "merchant.approved" ▼ Frontend Verification └─► Polls GET /api/v1/marketplace/merchant/status Check: onboarding_state = "APPROVED" ? 5. Error Handling Guide

HTTP Status Codes

Code Meaning Example Scenario
200 ✅ Success Onboarding step completed
400 ❌ Bad Request Invalid location data (missing postal_code)
401 ❌ Unauthorized Missing or invalid Bearer token
403 ❌ Forbidden Trying merchant onboarding without completing platform onboarding
409 ❌ Conflict Already in "accepted" connection state
500 ❌ Server Error Backend failure (retry with exponential backoff)

Common Error Responses

❌ Invalid Location Data
{ "error": { "code": "VALIDATION_ERROR", "message": "Validation failed", "details": [ { "field": "postal_code", "message": "postal_code must be at least 5 characters" } ] } }
❌ Platform Onboarding Required
{ "error": { "code": "PREREQUISITES_NOT_MET", "message": "Complete platform onboarding before merchant onboarding", "currentStatus": "incomplete" } }
❌ Unauthorized
{ "error": { "code": "UNAUTHORIZED", "message": "Invalid or missing Authorization header" } }

Retry Strategy

🔄 Recommended Retry Logic:
  • Transient Errors (5xx): Retry 3 times with exponential backoff (300ms, 600ms, 1200ms)
  • Rate Limit (429): Retry after Retry-After header
  • Client Errors (4xx): Do NOT retry, show error to user
  • Idempotency: Use idempotency_id in POST requests to safely retry
// Retry with exponential backoff async function withRetry(fn, maxAttempts = 3) { for (let attempt = 1; attempt <= maxAttempts; attempt++) { try { return await fn(); } catch (error) { if (attempt === maxAttempts || error.status < 500) { throw error; } const delay = 300 * Math.pow(2, attempt - 1); await new Promise(resolve => setTimeout(resolve, delay)); } } }

Expected: Should succeed (idempotent), not error
Verify: Progress unchanged, user can continue
Scenario 3: Permission Check - Merchant Before Platform
Setup: User platform onboarding incomplete
Action: Try POST /marketplace/merchant/onboard
Expected: 403 Forbidden error
Verify: Error message: "Complete platform onboarding first"
